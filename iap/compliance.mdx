## Checkout Presentation Methods

There are three ways to present a web checkout from an iOS app. Each has different trade-offs for user experience and App Review:

| Method | User Experience | App Review | Notes |
|---|---|---|---|
| **External Safari** | User leaves app to complete purchase | Explicitly allowed by Apple | Zero review friction; sanctioned method |
| **In-App Browser** (SFSafariViewController) | User stays in app; Safari engine | May face slower review | Better UX but not explicitly sanctioned |
| **Inline WebView** (WKWebView) | Embedded directly in app UI | May face slower review | Best UX but not explicitly sanctioned |

### External Safari Browser

The user taps a button in your app and is taken to Safari (or their default browser) to complete the purchase. After payment, a universal link brings them back to your app. Based on Apple's updated guidelines, this is the explicitly sanctioned method and typically carries zero App Review friction.

### In-App Browser (SFSafariViewController)

Opens the checkout page inside your app using Safari's rendering engine. The user stays within your app throughout the purchase flow. This provides a better user experience than external Safari but is not the explicitly sanctioned method — App Review may be slower or raise objections.

### Inline WebView (WKWebView)

The checkout is embedded directly into your app's UI as a native-feeling bottom sheet or full-screen view. This provides the best conversion rates and user experience. Like SFSafariViewController, this approach may encounter more App Review scrutiny than external Safari.

<Tip>
  ZeroSettle supports all three checkout methods via remote config on the dashboard. You can switch between them without shipping an app update — start with external Safari for the smoothest review, then switch to an in-app method once approved.
</Tip>

## US Geofencing

The SDK only offers external checkout to users on the US App Store storefront, per Apple Guideline 3.1.1(a) and 3.1.3 (US-only scope). Storefront is determined server-side via the user's App Store account region at the time of product fetch. Users outside the US see StoreKit-only UX even if the developer has BYOS connected.

## Apple Pay Domain Registration

Apple Pay on web checkout requires `stripe.PaymentMethodDomain.create()` per connected Stripe account. ZeroSettle registers this automatically during BYOS onboarding — no manual developer action required. The `country` parameter passed to `stripe.paymentRequest()` is resolved from the connected account's country and passed through the checkout URL as `merchant_country` so Apple Pay works for non-US merchants.

## App Review Scenarios

- **External Safari mode (default):** Explicitly sanctioned. Typically passes review on first submission.
- **SFSafariViewController / WKWebView modes:** Allowed under current guidelines but reviewer may ask about the flow. Keep the external-link disclosure sheet enabled — ZeroSettle provides it automatically.
- **Sandbox vs. Live key mismatch:** A sandbox Apple account with a live publishable key will produce StoreKit entitlements flagged `is_sandbox=True` that don't match live product catalogs. This is a developer testing artifact, not a production bug. See [troubleshooting](/iap/troubleshooting).
